State transition design

Posted on 2017-04-24 | In testing

A colleague asked me a question about state transition design technique while she was preparing for the ISTQB advanced test analyst exam. I always found this test design technique is useful when we have the system/application state changed from one to another.

I once tested an application feature which can monitor the customer’s purchase record. If the customer has purchased in 90 days then the customer will be classified as ‘loyal’ customer and the system will provide more access to live market data for the customer.
Customer status transits from loyal to disloyal and to loyal and to loyal again. This is a very simple example but in actual testing there are more conditions and status I need to consider.

The question is below

17 When an employee requests payment of a claim from his/her health insurance, this request follows the sequences of states shown in the diagram from being requested until it is closed and removed. If information is missing or changed, the claim may be moved back to an earlier state to prevent payment. There are the following restrictions: If a claim in state Accepted has been Closed it can only be restored to the same state Accepted. If a claim in state Activated has been Closed it can only be restored to state Activated.

Starting from Activated: what is the number of 0-switch transitions and what is the number of allowed 1-switch?

To find out N-switch we need to understand what is N-switch. N-switch is the measurement of test coverage. It refers to the length of N+1.

This question is simple except for one tricky condition.

Let’s find 0-swith first.

Find two lines or edges in total.

So if we count all the nodes on level 3(The root level is level 1). There are 12 1-switch transitions.
Then the tricky part is there are two conditions in the question.

If a claim in state Accepted has been Closed it can only be restored to the same state Accepted.

This condition we can ignore because we start from Activated state.

If a claim in state Activated has been Closed it can only be restored to state Activated.

This condition removed the path Activiated > Closed > Accepted. So we have 11 1-switch transitions.

In practise we use the following method to transform the state transition diagram to state transition tree.
Step1: Choose initial state or start state as root. The level of root is 1.

Step2: If the current tree level is K. Check all the nodes on this level from left to right. Find all the next state of the nodes.

Step3: Repeate step2 until the nodes on level K shows up on level J and J < K, then this node is the final node. There is no need to generate more children for this node. Or the node is the end state we don’t need to continue to generate too.

Step4: If we want to have deeper coverage we can simply add one level to the tree generated with the rules above.
